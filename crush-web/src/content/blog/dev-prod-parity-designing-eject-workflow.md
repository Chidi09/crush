One of the Twelve-Factor App methodologies is **Dev/Prod Parity**: keeping development, staging, and production environments as similar as possible.

Historically, this principle is why developers adopted Docker locally. The theory was: "If I run my database and app in the exact same Linux containers locally as I do in production, I will never hit 'it worked on my machine' bugs."

But in practice, this introduced a massive productivity trade-off on Windows: **1.5 GB of RAM overhead, slow directory mounts, and lagging hot-restarts** due to hypervisor boundaries.

At Crush, we asked: **Can we have native development speed locally, while maintaining 100% container parity in production?**

The answer is yes. We call it the **eject workflow**.

---

## The Philosophy: Native Dev, Containerized Prod

We believe that local development and production serve different goals:
* **In Development**: Speed, immediate feedback, and lightweight resource usage are paramount. You want hot-restarts to take milliseconds, and your databases to spin up instantly.
* **In Production**: Isolation, strict security boundaries, standard scaling, and repeatable deployment packages are critical.

Running containers locally on Windows is an expensive way to achieve parity. Instead, Crush allows you to **develop natively** using your host OS tools (Postgres, NodeJS, MSVC) and **eject to containers** only when it's time to test, share, or ship.

---

## How `crush eject` Works

When you have a project configured with Crush, you develop by simply typing `crush` in your console. Crush handles everything natively.

When your feature is complete and you are ready to prepare your deployment pipeline, you run:

```powershell
crush eject
```

Crush performs an automated export of your native stack configuration into standard OCI artifacts:

1. **Stack Parsing**: Crush looks at your auto-detected stack information (e.g. Node + Next.js, or Java + Spring Boot).
2. **Generates a `Dockerfile`**: It writes a high-quality, production-ready, multi-stage `Dockerfile` optimized for your specific framework. For example, for a Node app, it generates a multi-stage build that separates dependency installation, bundling, and a minimal runtime runner.
3. **Generates a `compose.yml`**: It translates your native database dependencies (like Postgres and Redis) into standardized, production-ready services in a standard `compose.yml` file.
4. **Appends a Safety Marker**: Crush appends a safety comment marker (`# Generated by Crush - Ejected`) to these files. This marker tells Crush: *"These are for production; continue running native tools locally unless explicitly instructed otherwise."*

---

## The Ejected Pipeline

Once you run `crush eject`, your project directory has everything a standard CI/CD platform (like GitHub Actions, GitLab CI, or Jenkins) expects.

You can now:
1. **Commit** the generated `Dockerfile` and `compose.yml` files.
2. **Build and Tag** OCI-compliant images in CI:
   ```bash
   docker build -t my-app:latest .
   ```
3. **Deploy** to any standard Linux server, VPS (Hetzner, DigitalOcean), Azure Container Instances, GCP Cloud Run, or AWS ECS.

```
Local (Windows Host):
[Crush CLI] ──> Starts Native DBs + Runs Host Dev Code (Sub-second starts, ~30MB RAM)

CI & Production (Linux Cloud):
[Git Push] ──> [GitHub Actions] ──> [Docker Image] ──> [AWS/Hetzner Linux VPS]
```

## The Benefits: The Best of Both Worlds

By adopting the Crush eject workflow, you no longer have to compromise:

* **No Resource Bloat**: Your laptop stays cool, your battery lasts longer, and you retain hundreds of megabytes of RAM for your IDE and browser.
* **Instant Feedback Loop**: Hot compiles are native and near-instant. No slow virtualized VM mounts lagging behind your file edits.
* **Deploy with Confidence**: You still ship standard, secure Linux containers to production, maintaining complete parity with modern cloud architectures.
